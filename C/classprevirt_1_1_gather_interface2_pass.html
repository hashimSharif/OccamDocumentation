Detailed Description

GatherInterface2Pass implements one phase of a three phase Inter-Module specialization process. The steps of the process are:

1) Gathering external references across all modules, so that the specific usages of functions outside the modules where they are defined can be captured. This collection is accomplished by the 'Pinterface2' pass implemented, in the previrt::GatherInterface2Pass class.

2) Given the specific usages, serialized into a file as a previrt::proto::ComponentInterfaceTransform object, the SpecializeComponentPass creates specialized versions of function in the modules where they have been defined.

3) Once the specialized functions have been created, the corresponding calls to the specialized versions, need to be rewritten, taking a fewer number of arguments. The call rewriting process is accomplished by the 'Prewrite' Pass implemented by the previrt::RewriteComponentPass class.

GatherInterface2Pass is registered as a Pass ,with the PassManager internal database by the name 'Pinterface2'. The Pass takes three arguments. The flag 'Pinterface2-output' followed by a file name specifies the output file for the interface description. The flag 'Pinterface2-function' is followed by a function name, specifying the externally called function in the module. The 'Pinterface2-entry' flag is followed by a file containing the functions called (function names). All the arguments are optional.

virtual bool previrt::GatherInterface2Pass::runOnModule ( Module &  M)
inlinevirtual

The GatherInterface2Pass:runOnModule gathers the external references and external function calls. The operation is completed in the following steps:

1) Extract the call graph of the Module.

2) Traverse through the set of global variables defined in the module and add all with external linkage to the list of references and calls in the interface object

3) If the entry functions are input to the Pass in a file. The functions are parsed from the file.

4) The set of entry functions, helps us to only traverse the relevant sections of the call graph.

5) All the call-sites found by traversing the call graph that are external function calls are logged into the interface object along with the specific set of arguments (required in the next phases, SpecializeComponentPass and RewriteComponentPass)

6) If an output file name is specified to the Pass, the interface object, is serialized to the file.

Note: The serialized file will be input to the SpecializeComponentPass (defined in the file SpecializeComponent.cpp)
